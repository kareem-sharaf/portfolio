Deep Dive: System Architecture

A closer look at how I approach complex backend problems

Umrain — Multi-Vendor Travel Marketplace

One booking engine. Six service types. Three payment gateways. One shared interface.

The Problem

A travel marketplace needed to support hotels, tours, flights, boats, car rentals, and events — each with completely different booking models, pricing structures, and availability logic. The traditional approach would mean 6 separate booking systems with duplicated payment logic, separate admin panels, and inconsistent APIs. Every new service type would take weeks to build from scratch.

Architecture Overview

Key Technical Decisions

Why Polymorphic Relationships over Single Table Inheritance?

STI would bloat the bookings table with nullable columns for each service type. Polymorphic relationships keep each service's data normalized while allowing unified queries through the Bookable interface. Adding a new service = one new model implementing Bookable, not a table migration.

Why Strategy Pattern for Payments?

Vendors operate in different regions: Stripe for US/EU, PayPal for global, Paystack for Africa. Each gateway is wrapped behind a PaymentProcessor interface. Adding a new payment gateway means implementing one new class — existing booking logic remains untouched.

Why a Separate Pricing Engine?

Hotels use per-night with seasonal rates. Tours use per-person with group discounts. Flights use dynamic fare classes. Extracting pricing into a dedicated engine with per-type calculators keeps booking logic clean and testable. Each calculator handles only its own pricing rules.

Core Architecture Pattern

// The Bookable Interface — every service type implements this
interface Bookable
{
    public function calculatePrice(BookingRequest $request): Money;
    public function checkAvailability(DateRange $range): bool;
    public function generateBookingReference(): string;
}

// Hotel implementation
class HotelRoom extends Model implements Bookable
{
    public function bookings(): MorphMany
    {
        return $this->morphMany(Booking::class, 'bookable');
    }

    public function calculatePrice(BookingRequest $request): Money
    {
        return $this->pricingEngine
            ->forDateRange($request->checkIn, $request->checkOut)
            ->applySeasonal($this->seasonal_rates)
            ->convertTo($request->currency)
            ->calculate();
    }
}

// BookingService — works with ANY bookable type
class BookingService
{
    public function create(BookingRequest $request): Booking
    {
        $bookable = $this->resolveBookable(
            $request->bookable_type,
            $request->bookable_id
        );

        if (!$bookable->checkAvailability($request->dateRange)) {
            throw new AvailabilityException('Selected dates unavailable');
        }

        $price = $bookable->calculatePrice($request);

        return DB::transaction(function () use ($bookable, $request, $price) {
            $booking = Booking::create([
                'bookable_type' => get_class($bookable),
                'bookable_id'   => $bookable->id,
                'user_id'       => auth()->id(),
                'total'         => $price->amount,
                'currency'      => $price->currency,
                'reference'     => $bookable->generateBookingReference(),
            ]);

            $this->paymentService->charge($booking, $request->gateway);
            $this->commissionService->calculate($booking);

            return $booking;
        });
    }
}

Challenges Solved

Multi-currency pricing with seasonal rates: Built a pricing pipeline that chains seasonal rate lookup → discount application → currency conversion, each step isolated and testable independently.
Commission calculation across payment gateways: Each gateway returns different fee structures. The commission engine normalizes fees before calculating vendor payouts, ensuring accurate revenue splitting regardless of payment method.
Real-time availability across time zones: Hotel availability is date-based, tour availability is slot-based, flight availability is seat-based. Unified the availability check behind the Bookable interface while preserving type-specific logic.

Results

6 service types running on one unified polymorphic booking engine
1 new class is all it takes to add a new service type to the platform
3 gateways integrated without modifying existing booking logic
Isolated pricing, payment, and commission — each testable independently

Performance Optimization

Measure first, optimize second. Never guess.

  • Query analysis with EXPLAIN ANALYZE before any optimization
  • N+1 detection through eager loading audit
  • Strategic denormalization where read-heavy patterns justify it
  • Background processing (Celery/Queue) for non-blocking operations
Real Example — E-Learning Progress Page:
Before 
# O(n×m) queries per page load
# For each student → for each course → calculate progress
for enrollment in student.enrollments.all():
    progress = calculate_progress(enrollment)
    # Multiple queries per student — scales poorly with data
After 
# Denormalized progress + Django signals
class Enrollment(Model):
    cached_progress = FloatField(default=0)

@receiver(post_save, sender=LessonCompletion)
def update_progress(sender, instance, **kwargs):
    enrollment = instance.enrollment
    enrollment.cached_progress = (
        enrollment.completions.count() /
        enrollment.course.lessons.count() * 100
    )
    enrollment.save(update_fields=['cached_progress'])
# 1 query per student — constant time regardless of data size
O(n×m) → O(1) Query complexity per student
N+1 → 1 Queries per page load
Signal-based Cache updates on write, not read

System Architecture

Separate what changes from what stays stable.

  • Service layer between controllers and models — controllers stay thin
  • Strategy pattern for interchangeable implementations (payments, notifications)
  • Event-driven architecture for cross-cutting concerns
  • Database migrations as first-class citizens (Flyway, Laravel migrations)

Security by Design

Security isn't a layer you add — it's how you build.

  • Separate auth endpoints per user type to prevent role manipulation
  • Ability-based tokens (not just role-based) for granular access control
  • Immutable records where business rules demand audit trails
  • WebSocket channel interceptors with JWT validation

// Medical records: immutable after finalization
@PreAuthorize("hasAuthority('WRITE_MEDICAL_RECORD')")
public MedicalRecord update(Long id, RecordDTO dto) {
    MedicalRecord record = repository.findById(id);
    if (record.isFinalized()) {
        throw new BusinessException(
            "Finalized records cannot be modified"
        );
    }
    return repository.save(mapper.update(record, dto));
}
// Once finalized, no modification at the application layer

Dual authentication prevents user-type manipulation. Finalized medical records are immutable — protecting patient data integrity at the business logic layer, not just the database.
